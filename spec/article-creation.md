## A. ヤメラボ記事システムの完全仕様

### A-1. ArticleData 型（完全版）

```typescript
interface ArticleData {
  meta: {
    slug: string;           // 必須: URL識別子（例: "bengoshi-osusume"）
    title: string;          // 必須: ページタイトル＆H1（SEO用、30-60字）
    description: string;    // 必須: メタディスクリプション（120-140字）
    category: ArticleCategory; // 必須: 4種類のいずれか
    publishedAt: string;    // 必須: YYYY-MM-DD形式
    updatedAt?: string;     // 任意: 更新日（SEO更新シグナル用）
    readingTime: number;    // 必須: 読了時間（分）
  };
  excerpt: string;          // 必須: 記事イントロ（100-150字、感情フック含む）
  faq: Array<{              // 必須: 4問以上（JSON-LD FAQPage自動生成）
    question: string;
    answer: string;         // 80字以上、具体的な数字を含む
  }>;
  sections: ArticleSection[]; // 必須: 本文セクション配列（8-12セクション推奨）
  relatedSlugs?: string[];  // 推奨: 関連記事2-3本（内部リンク設計）
}

type ArticleCategory =
  | "retirement-guide"  // 退職ガイド（手続き系）→ blue
  | "money"             // お金の知識（税金・保険）→ emerald
  | "work-rights"       // 労働者の権利（有給・法律）→ purple
  | "taishoku-daikou";  // 退職代行 → orange

interface ArticleSection {
  id: string;           // 必須: アンカーリンク用（英数字ハイフン、例: "three-types"）
  heading: string;      // 必須: H2見出し（装飾不可、プレーンテキストのみ）
  content: string;      // 必須: 本文段落（\nで複数段落分割、**太字**/__下線__対応）
  list?: string[];      // 任意: 箇条書き（•記号付き）
  numberedList?: string[]; // 任意: 番号付きリスト（丸バッジ番号）
  table?: ArticleTable; // 任意: 比較表（必ず構造化データで渡す）
  timeline?: ArticleTimeline; // 任意: タイムライン（手順・フロー）
  callout?: ArticleCallout;   // 任意: 注釈ボックス（1セクションに1つ）
  checklist?: ArticleChecklist; // 任意: チェックリスト（確認事項・選択基準）
  blockquote?: string;  // 任意: 引用（プレーンテキストのみ、装飾不可）
  subSections?: Array<{ // 任意: H3サブセクション
    id: string;
    heading: string;
    content: string;    // 必須: **太字**/__下線__対応。ただし短く（150字以内推奨）
    list?: string[];    // 推奨: サービス特徴・メリット・注意点を箇条書きで整理
    callout?: ArticleCallout; // 推奨: そのサブセクション固有のポイントを強調
  }>;
}
```

**subSections のレンダリング（BOXILカードスタイル）:**
- H3見出しはカード外に配置。primary色・太字・下にボーダーライン
- カード本体は `1px solid #e0e0e0`、padding 24px、border-radius/shadow なし
- カード内: content（概要テキスト）→ list（divide-y区切り）→ callout の順
- BOXILマガジン（https://boxil.jp/mag/）のサービス比較カードUIをベースに設計

**⚠️ subSections 使用時の鉄則（文字の壁を防ぐ）:**
- subSections の content だけで200字を超えてはならない（超える場合は list/callout で情報を分離する）
- 各 subSection に最低1つの構造化要素（list または callout）を含めること
- 「content に全情報を詰め込む」のではなく「content = 1〜2文の概要、list = 詳細情報」という分割を徹底
- subSection が4つ以上連続する場合は、親セクションにも list/table を置いて「全体像 → 各論」の流れを作る

```typescript
interface ArticleTable {
  headers: string[];    // **太字**/__下線__対応
  rows: string[][];     // セル内**太字**/__下線__対応
  caption?: string;     // プレーンテキストのみ
}

interface ArticleTimeline {
  items: Array<{
    time: string;       // タイムスタンプ（例: "STEP 1", "即日"）プレーンテキスト
    title: string;      // プレーンテキスト
    description?: string; // プレーンテキスト
  }>;
}

interface ArticleCallout {
  type: "tip" | "warning" | "info"; // tip=緑, warning=黄, info=青
  content: string;      // **太字**/__下線__対応
}

interface ArticleChecklist {
  items: string[];      // プレーンテキストのみ（装飾不可）
}
```

### A-2. レンダリング順序（固定）

```
H2見出し（heading）
  ↓ content（段落テキスト）
  ↓ list（箇条書き）
  ↓ numberedList（番号付きリスト）
  ↓ table（比較表）
  ↓ timeline（タイムライン）
  ↓ callout（注釈ボックス）
  ↓ checklist（チェックリスト）
  ↓ blockquote（引用）
  ↓ subSections（H3サブセクション群。各subSection内: content → list → callout の順）
```

### A-3. ⚠️ 絶対に守る制約（違反すると表示が壊れる）

| やってはいけないこと | 正しい書き方 |
|---|---|
| contentにMarkdownテーブル記法 `\| 列 \|` を書く | `table` フィールドを使う |
| contentに箇条書き `- 項目` を書く | `list` フィールドを使う |
| contentに番号リスト `1. 項目` を書く | `numberedList` フィールドを使う |
| contentに `# 見出し` を書く | `heading` フィールドを使う |
| subSectionsのcontentに全情報を詰め込む（200字超のテキスト壁） | contentは1〜2文の概要のみ。詳細はlist/calloutに分離 |
| blockquoteに `**太字**` を書く | プレーンテキストのみ有効 |
| checklistのitemsに `**太字**` を書く | プレーンテキストのみ有効 |
| timeline.items.*に `**太字**` を書く | プレーンテキストのみ有効 |
| table.captionに `**太字**` を書く | プレーンテキストのみ有効 |

## B. 退職代行記事 タイプ別テンプレート

### タイプ1: ランキング・比較記事（例: 退職代行 弁護士 おすすめ）

**想定文字数**: 4,500〜6,000字
**想定セクション数**: 8〜10

```
excerpt: 共感 → この記事でわかること（3点） → 結論先出し（「最おすすめはXX」）

sections:
  1. id: "moumuri-incident"（必須）← 2026年2月モームリ逮捕後のヤメラボ追加項目
     heading: 退職代行選びで失敗しないために：2026年2月の変化
     content: 逮捕事件概要 → 「今も使って大丈夫か」への回答
     callout: warning「民間業者の交渉行為は弁護士法72条違反」

  2. id: "three-types"
     heading: 退職代行3種類の違いと権限

  5. id: "service-detail"（H2 + subSections H3で各社詳細）
     heading: 各サービス詳細レビュー
```

## C. 品質設計原則（4段階）

### C-1. 意図分析（Intent Analysis）

| KWパターン | 意図タイプ | 最適な記事形式 |
|---|---|---|
| 退職代行 弁護士 おすすめ | Commercial(C) | ランキング + 比較表 + AF誘導 |
| 退職代行 料金 相場 | Informational(I) | 料金表 + 解説 + 結論 |
| 退職代行 後悔 | I + 感情 | 共感 + 失敗事例 + 解決策 |
| 退職 言えない | I + 不安 | 心理的共感 → 解決策提示 |
| 退職代行 新卒 | C + 不安 | 新卒特有リスク + サービス推薦 |
| 退職代行 バイト | C | バイト向け料金 + 手順 |

### C-2. 特化調査（Specialized Research）

**記事作成前に必ずやること:**
1. 対象KWでGoogle検索 → 上位5件のH2見出し一覧を記録
2. PAA（他の人はこちらも質問）欄の質問を全部記録 → FAQに使う
3. 1位記事の推定文字数
4. 個人ブログが上位10件に入っているか確認（KD判定）
5. 競合記事が触れていない情報を1つ以上発見

**退職代行記事の場合、必ず確認する独自情報:**
- 掲載するサービスの料金（公式サイトで最新情報を確認）
- 返金保証の有無と条件（公式サイトで確認）
- モームリ逮捕事件の最新情報（2026年2月時点での続報）
- AF先サービスがヤメラボの既存AFリンクに含まれるか確認

### C-3. 適応構成（Adaptive Structure）& スキャナビリティ

**✓ テキストだけのセクションが3つ以上連続してはならない**
必ずtable/checklist/callout/list/timeline等の構造化要素で視覚的に区切ること。

**✓ 200字を超える連続テキストブロックは禁止**
content が200字を超える場合は `\n` で段落を分割するか、情報の一部を list/numberedList に移す。
subSections の content も同様。長文テキスト＝読者が離脱する最大の原因。

**✓ 各 subSection に最低1つの構造化要素（list or callout）を含めること**
subSection が content のみ = 文字の壁 → 読者は読み飛ばす。
必ず list（特徴・料金・向いているケース等）または callout（ポイント・注意点）を付ける。

**セクションの情報量バランス:**
- 導入・まとめ: content 200〜300字（短くてOK）
- メインセクション: content 400〜600字 + 構造化要素
- subSections併用セクション: 親content 300字以上 + list（全体像の提示）
- 各subSection: content 150字以内 + list（3〜5項目）+ callout（推奨）

### C-4. 専門執筆（Expert Writing）

**具体性の基準（これを守らないとYMYL基準を満たせない）:**

| ✗ 抽象的（禁止） | ○ 具体的（必須） |
|---|---|
| 「料金が安い」 | 「一律19,800円（税込）、業界最安値クラス」 |
| 「弁護士に相談すると安心」 | 「弁護士法72条で非弁行為は2年以下の拘禁刑または300万円以下の罰金」 |
| 「成功率が高い」 | 「退職成功率100%（2026年3月時点、同社公称）」 |
| 「モームリが問題になった」 | 「2026年2月3日、株式会社アルバトロス代表・谷本慎二らが弁護士法違反で逮捕」 |
| 「料金は後払い可能」 | 「退職決定後7日以内の支払いでOK」 |

**一文一義の原則:**
- 1文に1つの情報のみ
- 接続詞「〜であり、〜でもあります」型の複文は分割する
- 読点3つ以上の長文は禁止

**結論ファースト:**
- 各セクションの最初の文で「このセクションの結論」を述べる
- 詳細は後続の段落・リストで展開

## D. YMYL・退職代行固有の品質基準

### D-1. 法的根拠の引用ルール（全退職代行記事で必須）

**弁護士法第72条（非弁行為禁止）:**
> 「弁護士又は弁護士法人でない者は、報酬を得る目的で訴訟事件、非訟事件及び審査請求、再調査の請求、再審査請求等行政庁に対する不服申立事件その他一般の法律事件に関して鑑定、代理、仲裁若しくは和解その他の法律事務を取り扱い、又はこれらの周旋をすることを業とすることができない。」
> ※出典: e-Gov法令検索 弁護士法第72条

**弁護士法第77条（非弁行為の罰則）:**
> 「二年以下の拘禁刑又は三百万円以下の罰金に処する。」

**民法第627条第1項（退職の自由）:**
> 「当事者が雇用の期間を定めなかったときは、各当事者は、いつでも解約の申入れをすることができる。この場合において、雇用は、解約の申入れの日から二週間を経過することによって終了する。」

**引用の実装方法:**
- `content` にインライン引用: `民法627条1項では「退職はいつでも申し入れできる」と定められており...`
- `table.caption` に出典: `※出典: e-Gov法令検索`
- `blockquote` に条文の重要部分を抜粋

### D-2. モームリ逮捕事件への対応（2026年2月時点）

すべての退職代行記事の**冒頭または第1セクション**に以下を盛り込む:

```
事件概要: 2026年2月3日、退職代行サービス「モームリ」を運営する
株式会社アルバトロスの代表・谷本慎二らが弁護士法違反（非弁提携）の
疑いで逮捕された。

違法内容: 弁護士を紹介する見返りとして1件あたり約1万6,500円の
報酬を受け取っていたとされる（弁護士法27条・72条違反）。

現在の状況: 新規受付を一時停止。

利用者への影響: 退職代行そのものは違法ではない。
ただし、運営元の選択が重要。弁護士・労働組合型であれば安全に利用できる。
```

### D-3. AF（アフィリエイト）表記のルール

- ページ最上部に「本ページはプロモーションを含みます」と記載（ページ側で自動挿入される）
- AFリンクにはすべて `rel="nofollow sponsored"` 属性（実装済み）
- サービス情報は掲載時点の公式サイト情報に基づき記載
- `※2026年○月時点の情報です` をtable.captionまたはcallout.infoで明記

### D-4. 事実確認チェック（料金・実績は必ず公式サイトで確認）

| 確認項目 | 確認方法 |
|---|---|
| 各サービスの料金 | 公式サイト（最新の料金ページ） |
| 返金保証の有無・条件 | 公式サイトの「よくある質問」 |
| 退職成功率 | 公式サイト（「○%」の根拠が記載されているか確認） |
| 対応時間（24時間・即日） | 公式サイトの「サービス内容」 |
| 顧問弁護士の実在確認 | 日本弁護士連合会「ひまわりサーチ」で検索 |

## E. 装飾要素の最低基準（全記事で必須）

| 要素 | 最低数 | 注意事項 |
|---|---|---|
| `table` | 1つ以上 | 必ず比較表を入れる。contentにMarkdownテーブルは書かない |
| `checklist` | 1つ以上 | 選択基準 or 確認事項。汎用コピペ禁止、記事固有の内容で |
| `callout` | 1つ以上（warning推奨） | 退職代行は法的リスクが多い → warning中心 |
| `list` | 2つ以上 | 特徴・メリット・注意点など |
| `blockquote` | 1つ以上 | 専門家的コメント・重要な結論を引用形式で強調 |
| 連続テキスト | 3セクション以上禁止 | 構造化要素で視覚的リズムを作る |

## F. 文字数・品質数値基準

| 要素 | 基準 | 根拠 |

| 記事全体（ランキング記事） | 4,500字以上 | 競合上位が5,000〜8,000字 |
| 記事全体（情報・解説記事） | 3,000字以上 | 情報密度重視 |
| 各セクション content | 300字以上（導入・まとめ以外は500字推奨） | 薄い内容をなくす |
| subSections併用セクションの親content | 300字以上 + list必須（全体像提示） | 親が空洞化しないようにする |
| 各subSectionのcontent | 150字以内（概要のみ。詳細はlistに分離） | 文字の壁を防ぐ |
| excerpt | 100〜150字（感情フック含む） | 一覧カード・SNSシェア用 |
| FAQ各回答 | 80字以上（具体的な数字を含む） | JSON-LD / Featured Snippet |
| FAQの問数 | 4問以上 | FAQPage スキーマ |
| 太字・下線の使用頻度 | 1セクションに2〜3箇所 | 多用すると効果が薄れる |

## G. SEO品質基準

### G-1. タイトル設計

- メインKWを先頭30文字以内に配置
- 数字を含める（「3選」「5つ」「2026年最新」）
- 60字以内（検索結果での切れを防ぐ）
- 例: `退職代行 弁護士おすすめランキング5選【2026年最新】費用・選び方も解説`

### G-2. メタディスクリプション

- 核心情報を先頭70字以内に（モバイル検索での切れ対策）
- 全体120〜140字
- KWを自然に含む
- 例: `弁護士が運営する退職代行のおすすめを厳選比較。料金・成功率・返金保証を一覧表で解説。モームリ逮捕後に安全な弁護士型を選ぶポイントも紹介。`

### G-3. H1 / H2 / H3 の構造

- H1: 記事タイトルと同一、ページ内1つ（自動設定）
- H2: 各sectionのheading（7〜10個）
- H3: subSectionsのheading（各H2に2〜4個）
- H4以下: 使用禁止

### G-4. 内部リンク

- 3〜10本の内部リンク
- アンカーテキストにKWを含む（例: 「退職代行の選び方」「失業保険の計算はこちら」）
- `relatedSlugs` に2〜3本の関連記事を設定
- 公開後に既存記事のrelatedSlugsからも新記事へリンクを追加

### G-5. 構造化データ（自動生成）

- Article: slug, title, description, publishedAt, updatedAt
- FAQPage: faqフィールドから自動生成
- BreadcrumbList: URLパスから自動生成

## H. マルチエージェント記事作成フロー

### H-1. エージェント役割一覧

| エージェント | 役割 | 主な出力物 |

```
  ↓
[PM] アウトラインをレビュー → 問題あれば修正指示 → 合格したらWriterへ
  ↓
[Writer] TypeScript記事データファイルを作成（本仕様書を参照）
  ↓
[QA] 独立したコンテキストでQAチェック実行
  ↓
[PM] QAレポートをレビュー
  ↓
  ├─ 問題あり → [Writer]に修正指示 → 再QA（問題ゼロまで繰り返す）
```

## I. 文字数計測の手順（QAエージェント用）

TypeScriptファイルから文字数を正確に測定する方法:

```

3. list の各項目 + numberedList の各項目も文字数に加算
4. table の headers + rows の全セルも加算
5. excerpt + 各 FAQ の answer も加算

目安計算:
  - ランキング記事: content合計 + list合計 + table合計 ≥ 4,500字
  - 解説記事: 同上 ≥ 3,000字

⚠️ heading（H2見出し）は記事の文字数に含めない
⚠️ relatedSlugs / meta フィールドも含めない
```

## L. 失敗パターンと回避方法

| 失敗パターン | 症状 | 回避方法 |
|---|---|---|
| contentにMarkdown記法を書く | 表・箇条書きが単なる段落テキストになる | 必ず構造化フィールド（table/list）を使う |
| checklistに**太字**を書く | 画面に `**テキスト**` がそのまま表示される | checklist.itemsはプレーンテキストのみ |
| subSectionsにtable/calloutを書く | 型エラーまたは表示されない | 独立したsectionに昇格させる |
| QAなしでデプロイ | 品質問題が本番に出る | 必ずQAエージェントを通す |
| 料金を確認せず記載 | 古い料金が掲載される | 公式サイトで最新料金を確認（毎記事） |
| relatedSlugsに実在しないslugを書く | ビルドは通るが関連記事が表示されない | index.tsのslug一覧と照合 |
| FAQが3問以下 | FAQPageスキーマが無効になる | 必ず4問以上 |
| 抽象的な表現のまま | YMYL評価が下がる | 全「安い」「多い」「高い」に数字を追加 |
| subSectionsのcontentに全情報を詰め込む | H3セクションが「文字の壁」になり読者が離脱 | contentは150字以内の概要のみ。詳細はlist/calloutに分離 |
| subSections併用セクションの親contentが空洞 | H2直下が1行だけで薄い印象 | 親contentも300字以上。list で各社の「向いている人」を一覧提示 |
| listが記事全体で1つしかない | 装飾要素の最低基準（E）を満たさない | Writer段階で list ≥ 2 を意識的に確認 |
| spec上の料金で記事を書く（公式未確認） | 古い料金が掲載される | specの料金は参考値。必ず公式サイトで最新を確認 |
